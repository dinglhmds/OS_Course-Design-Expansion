# 操作系统课程设计实验报告

## CPU 调度仿真框架的设计与实现

---

## 一、实验目的

1. **理解 CPU 调度算法的基本原理**：通过代码实现深入理解 FCFS、RR、Priority、SJF、EDF、RMS 等经典调度算法的核心机制。
2. **掌握离散事件仿真方法**：学习如何用时间片步进的方式模拟操作系统调度器的行为。
3. **对比分析不同调度算法的性能差异**：在相同任务流下，量化比较各算法的周转时间、公平性、截止期限满足率等指标。
4. **研究实时调度与多核调度**：扩展基础框架，支持周期性实时任务和多核并发调度场景。
5. **培养系统级编程能力**：运用 Python 实现完整的仿真系统，包括接口设计、引擎实现、指标计算和可视化输出。

## 三、实验内容

### 3.1 系统架构设计

本项目采用**插件式架构**，核心组件包括：

| 模块 | 职责 |
|:---|:---|
| `interfaces.py` | 定义 `Task` 数据类和 `BaseScheduler` 抽象基类 |
| `engine.py` | 离散事件仿真引擎，支持 Batch/Realtime/Multicore 三种模式 |
| `metrics.py` | 性能指标计算（周转时间、公平性、截止期限错失率、加速比等） |
| `report_generator.py` | CSV 报告导出 + Matplotlib 甘特图生成 |
| `main.py` | CLI 入口，支持三种仿真模式的参数配置和自动对比 |
| `algorithms/` | 调度算法实现目录 |

### 3.2 实现的调度算法

| 算法 | 类型 | 核心策略 |
|:---|:---|:---|
| **FCFS** | 批处理/非抢占 | 先来先服务，FIFO 队列 |
| **RR** | 批处理/抢占 | 时间片轮转，quantum 到期强制切换 |
| **Priority** | 批处理/非抢占 | 按优先级排序，高优先级优先执行 |
| **SJF** | 批处理/非抢占 | 短作业优先，按 burst_time 排序 |
| **EDF** | 实时/抢占 | 最早截止期限优先，动态优先级 |
| **RMS** | 实时/抢占 | 速率单调调度，周期越短优先级越高（静态） |
| **GlobalRR** | 多核/抢占 | 全局队列 + 多核时间片轮转 |

### 3.3 三种仿真模式

#### 3.3.1 Batch 模式
- 任务一次性到达，全部完成后仿真结束
- 关键指标：平均周转时间、平均等待时间、公平性指数、吞吐量

#### 3.3.2 Realtime 模式
- 任务按周期自动生成新实例（Job）
- 引擎检测截止期限违规：`completion_time > absolute_deadline`
- 关键指标：截止期限错失率、最大响应时间、抖动

#### 3.3.3 Multicore 模式
- 支持多 CPU 核心并行执行
- 迁移惩罚：任务跨核切换时 `remaining_time += 1`，模拟缓存失效开销
- 关键指标：加速比、迁移次数

python main.py --mode realtime --duration 60 --quantum 4

**结果分析**：

| 算法 | 平均周转时间 | 平均等待时间 | Jain 公平指数 | 特点 |
|:---|:---|:---|:---|:---|
| **SJF** | **12.38** ⭐ | **8.25** ⭐ | 0.5522 | 短作业优先，整体最优 |
| RR | 15.88 | 11.75 | **0.7071** ⭐ | 最公平，但切换开销大 |

- SJF 在平均周转时间和等待时间上表现最优，因为短作业优先策略减少了长任务对短任务的阻塞
- RR 的公平性指数最高（0.7071），时间片轮转保证了各任务相对均衡的 CPU 时间分配
- Priority 算法公平性最差（0.5737），因为高优先级任务可能垄断 CPU
- 所有算法吞吐量相同（0.2424），因为总任务数和总时间固定

### 5.3 Realtime 过载实验结果（关键测试）

**原始输出**：

```
Realtime OVERLOAD Test (133% utilisation)

[FCFS] 22 jobs, 15 missed, rate=68.18%
[  RR] 22 jobs, 18 missed, rate=81.82%
[ EDF] 22 jobs, 15 missed, rate=68.18%
[RMS] 22 jobs, 3 missed, rate=13.64%

Algo    Jobs  Miss   MissRate  MaxRT  AvgRT  Jitter
---------------------------------------------------
FCFS      22    15     0.6818     22  13.25    4.84
RR        22    18     0.8182     30  17.77    6.27
EDF       22    15     0.6818     25  12.88    5.58
RMS       22     3     0.1364     10   5.33    3.30
```

**结果分析**：

| 算法 | 截止期限错失率 | 最大响应时间 | 平均响应时间 | 分析 |
|:---|:---|:---|:---|:---|
| **RMS** | **13.64%** ⭐ | **10** ⭐ | **5.33** ⭐ | 静态优先级保护短周期任务 |
| EDF | 68.18% | 25 | 12.88 | 过载时动态优先级无法区分关键任务 |
| FCFS | 68.18% | 22 | 13.25 | 完全不考虑截止期限 |
| RR | **81.82%** 💀 | 30 | 17.77 | 时间片切换浪费严重，最差 |

**重要发现**：
- **RMS 碾压级优势**：在 133% 过载情况下，仅 13.64% 的任务超时，因为静态优先级确保了短周期（高优先级）任务得到保障
- **EDF 退化**：理论上 EDF 是单核最优动态优先级算法，但在严重过载时，所有任务同时竞争，动态优先级无法有效区分
- **RR 完全不适合实时场景**：时间片轮转完全不考虑截止期限，导致 81.82% 的任务超时
- FCFS 和 EDF 表现相近（68.18%），说明在极端过载下，简单的 FIFO 和复杂的动态优先级效果相当

## 六、关键设计亮点

### 6.1 向后兼容的接口设计

`BaseScheduler.schedule()` 默认实现调用 `next_task()`，使得原有单核算法无需修改即可在新引擎上运行：

```python
def schedule(self, current_time, cpu_status_dict):
    task = self.next_task(current_time)
    return {0: task}
```

### 6.2 迁移惩罚机制

多核模式下，任务跨核切换时自动施加惩罚：

```python
if assigned.last_cpu != -1 and assigned.last_cpu != cpu_id:
    assigned.remaining_time += 1  # 模拟缓存失效开销
    assigned.migration_count += 1
```

### 6.3 甘特图多格式自适应

`report_generator.py` 自动检测 gantt 数据格式，支持单核/多核、批处理/实时四种组合。

## 七、遇到的问题与解决方案

| 问题 | 原因 | 解决方案 |
|:---|:---|:---|
| 实时模式下 RMS 报告 0 miss 但任务未完成 | 仿真结束时未检查未完成任务是否已超截止期限 | 在引擎末尾增加 post-simulation cleanup，标记 `completion_time is None and current_time > absolute_deadline` 的任务为 missed |
| 多核甘特图 KeyError | `_detect_gantt_format()` 只检查第一个非空 entry，未考虑多核 | 扫描所有 entry 取最大 cpu_id |
| 单核算法适配新接口 | `schedule()` 签名从 `(current_time, last_running_task, last_task_run_time)` 改为 `(current_time, cpu_status_dict)` | 更新 EDF/RMS 的 `schedule()` 方法，返回 `{0: task}` |

## 八、实验总结

### 8.1 主要收获

1. **深入理解了调度算法的本质差异**：
   - 批处理算法关注平均周转时间和公平性
   - 实时算法关注截止期限满足率
   - 多核算法关注负载均衡和缓存亲和性

2. **验证了理论结论**：
   - SJF 在平均周转时间上最优（但需预知 burst_time）
   - RR 最公平但切换开销大
   - RMS 在过载时比 EDF 更稳健（静态优先级的可预测性）

3. **掌握了仿真框架设计方法**：
   - 插件式架构便于扩展新算法
   - 离散事件模拟适合教学和理解
   - 统一指标便于公平对比

### 8.2 可扩展方向

1. **能耗模型**：增加 CPU 频率调节，模拟 DVFS 节能调度
2. **缓存模拟**：更精细的缓存层次模型
3. **优先级继承**：实现实时系统的优先级天花板/继承协议
4. **ML 调度器**：用强化学习训练调度策略
